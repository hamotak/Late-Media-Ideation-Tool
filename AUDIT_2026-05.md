# AUDIT — Eric YT Channel AI

Date: 2026-05-17
Scope: every route under `src/app/`, every API route under `src/app/api/`,
`src/lib/chat-tools.ts`, key shared components. Read-only audit — no source
files were modified.

The audit is opinionated. Anything that's working fine and on-spec is
skipped. The goal is to surface stuff HAmo can decide to cut, fix, or
escalate before the next prompt.

## Section A — Critical bugs (real broken behavior)

### A1. Sidebar "Settings" link silently hides 4 of the 5 settings tabs

File: `src/components/sidebar.tsx:61`
Layout: `src/app/settings/layout.tsx:13-19`

The sidebar has one item, `Settings → /settings`, which lands on the
**Preferences** tab (theme toggle only — `src/app/settings/page.tsx`). The
sub-tabs **Integrations / Import / Logs / Alerts** are only reachable AFTER
the user clicks Settings AND notices the tab bar above the theme card.

From the brief: "Settings (with tabs: Preferences/Integrations/Import/Logs/
Alerts)". The tabs exist (`SettingsLayout`), but a first-time user clicking
"Settings" sees just a theme picker and can rationally conclude there's
nothing else. The Connect Banner pushes them to `/settings/integrations`
which is fine, but every other entry (Logs, Alerts, Import) is buried.

Real bug because **Alerts** is the only place to wire up Telegram + the
poll cron — non-discoverable for the target user.

### A2. Tutorial page contradicts itself on what's shipped

File: `src/app/tutorial/page.tsx`

- Line 90 (quick-start step 4): `<ComingSoon />` pill in front of
  "Outliers" — implies it's not shipped.
- Line 171 (the Pages list, same file): Outliers shown with
  `<ActivePill />` status="active".
- Line 94 (quick-start step 5): "Topic Ideation" tagged "Coming soon"
  inline, BUT the Pages list (line 201) describes it as available
  *now* in AI Chat: "Available now in AI Chat — full standalone page
  comes later."

A non-technical user reading top-to-bottom is told Outliers is coming
soon, then 80 lines later told it's active. This is the tutorial — it
needs to be authoritative. Fix: drop the `<ComingSoon />` pill from step
4 (it shipped). The Topic Ideation hedge is fine *if* worded
consistently — currently it's both "coming soon" and "use it in chat
now". Pick one.

### A3. Tutorial "Pages list" omits **Chat** from the quick-start

File: `src/app/tutorial/page.tsx`, quick-start ordered list (lines 63-98).

Quick-start ends at step 5 (Topic Ideation, marked coming soon). It
never tells the user "use Chat". Chat is the central ideation agent —
the brief explicitly says so — but the onboarding sequence routes the
user to a page that's not built yet. A new user would never discover
Chat from the tutorial.

### A4. `Channel Info` page mixes server-active and `?focus=` channel sources, causing the detail widgets to show the wrong channel's data when the user clicks a row in the "All channels" table

File: `src/app/channel-info/page.tsx:256-269`

`SingleChannelCard` passes `channel.channelId` (the focused channel) to
`/api/channel?channelId=...` — good. But the page-level `ChannelAudience`
and `ChannelRevenue` components (lines 455-456) take **no `channelId`
prop**. They read the server-side *active* channel pointer.

So when the user is in `viewMode === "all"` and clicks a row to
`?focus=<otherId>`:
- AboutCard, MetaCard, ThemesCard, TranscriptsCoverageCard render
  data for the focused row (correct).
- ChannelAudience + ChannelRevenue render data for whichever channel
  the global active pointer says (often a different one).

User sees a header card for "Channel A" with audience/revenue numbers
from "Channel B". This is a real cross-channel data leak in the UI.

### A5. `/competitors` "Add Competitor" tier select shows the tooltip on the trigger but not on options

File: `src/app/competitors/page.tsx:600-611`

The `<option title={TIER_TOOLTIP[t]}>` on `<option>` elements is silently
ignored by every browser — `title` on `<option>` doesn't render a
tooltip natively. The dropdown shows just the tier name with no
explanation. A new user picks "Authority" / "Breakthrough" / "Adjacent"
/ "Far" with zero context for what they mean.

Same issue at `src/app/competitors/[id]/page.tsx:334`,
`src/app/competitors/page.tsx:1379-1383`,
`src/app/competitors/page.tsx:1494-1498`.

The tier pill tooltip on the card itself *does* work
(`title={TIER_TOOLTIP[t]}` on a span). Just the select options that fail.

### A6. `/competitors` "Add Competitor" wedges if no `activeId` but only surfaces error after submit

File: `src/app/competitors/page.tsx:327-362`

Button enabled state checks `!activeId` but the input is enabled. User
types, hits Enter (line 596), `addCompetitor` rejects with "No active
channel — set one from the top-right channel picker." Better UX: gray
out the entire add row with a banner "Pick an active channel first."

### A7. `Re-fetch captions` button on a video with `captionsUnavailable: true` is wired to retry but the empty-state's primary button is locked off

File: `src/app/videos/[id]/page.tsx:660-686` and lines 772-786.

In the empty state (no transcript), once `captionsUnavailable` flips
true, the "Get YouTube captions" button is `disabled`. In the
has-transcript state the same path is exposed as "Re-fetch captions"
WITHOUT honoring `captionsUnavailable`. Two different code paths gating
the same network call differently. If captions become available later
(YouTube uploaded auto-captions), the user can only retry from the
has-transcript path — which they can't reach because there's no
transcript yet.

### A8. `Channel Info` Channels API doesn't include `description`, so AboutCard is fed `description: null` hard-coded

File: `src/app/channel-info/page.tsx:275`

```ts
const detailChannel = {
  id: channel.channelId,
  title: channel.title,
  handle: channel.handle,
  description: null, // /api/channel-info doesn't include description today
  imported_at: 0,
};
```

`imported_at: 0` and `description: null` are hard-coded — AboutCard
will always render the empty/zero state for these fields. Either fix
the API to include them or strip them from AboutCard's contract. As
shipped, AboutCard is partially broken on /channel-info.

### A9. `useMemo` used like `useEffect` to re-sync rule draft in AlertsPage

File: `src/app/settings/alerts/page.tsx:579-581`

```ts
useMemo(() => {
  setDraft(rule);
}, [rule]);
```

`useMemo` is for computing values; using it for a side effect (calling
`setDraft`) is anti-pattern and may not run consistently across React
versions. Should be `useEffect`. Result: editing a rule and getting a
parallel refresh from another row may leave stale "dirty" state on
screen.

### A10. Send-test handler for Telegram bot uses HTTP `PUT` to `/api/alerts/config`

File: `src/app/settings/alerts/page.tsx:199`

```ts
const res = await fetch("/api/alerts/config", { method: "PUT" });
```

If the route handler doesn't export `PUT`, the click silently 405s.
Worth double-checking the route file
(`src/app/api/alerts/config/route.ts`) exports a `PUT` handler. If
not — broken button.

## Section D — Agent data inventory

What follows is every tool currently registered in `src/lib/chat-tools.ts`.
For each tool I list the data it can actually read.

### D.1 What the agent CAN access today

**Group `youtube`** (lines 71-166, dispatcher 511-620)
- `channel_summary` — active channel row + video stats (`channels`, `videos` tables, active-channel scoped).
- `list_my_videos` — `videos` table rows (id, title, views, likes, comments, duration, publishedAt). Active-channel scoped.
- `search_my_transcripts` — FTS over `transcripts.text` for active channel.
- `get_video_comments` — LIVE YouTube Data API top-level comments by video ID (paid quota).
- `list_video_comments_cached` — cached `comments` rows for one of the user's videos.
- `search_my_comments` — FTS5 across all cached comments on user's videos.
- `get_comment_thread` — one comment + its replies from cache.
- `search_youtube` — public YouTube Data API search (100 quota units / call).

**Group `analytics`** (lines 200-252, dispatcher 654-718)
- `execute_sql` — read-only SELECT over the local SQLite schema (`SQL_SCHEMA` in `src/lib/sql-tool.ts`). 200 row cap.
- `youtube_trending` — region/category trending list via YouTube Data API.
- `niche_explorer` — top-5 channels + top-10 outlier videos for a topic via the YouTube API (~200 quota units).
- `fetch_transcript` — any public YouTube video's captions, free (yt-dlp + timedtext / Innertube), cached in `transcripts`.

**Group `research`** (lines 254-277)
- `youtube_suggest` — YouTube autocomplete via the public completion endpoint. Free, keyless.
- (NOTE: Google Trends tools were removed — they 429'd consistently. See comment lines 269-276.)

**Group `exa`** (lines 168-198)
- `web_search` — Exa semantic web search (titles + URLs + snippet text).
- `web_fetch` — Exa cleaned-text reader for up to 5 URLs.

**Group `apify`** (lines 279-309)
- `scrape_youtube_channel` — Apify YouTube channel scrape — videos, views, likes, duration, optional transcripts.
- `get_youtube_transcript` — Deepgram-backed transcript (yt-dlp pulls audio locally, streams to Deepgram, caches in `transcripts`).

**Group `yt_analytics`** (lines 321-402, dispatcher 786-845)
- `get_channel_analytics_overview` — channel totals + period deltas + daily series + top 10 videos (via OAuth).
- `get_video_analytics` — per-video deep bundle (retention curve, traffic sources, playback locations, top search terms that reached the video, sharing services, OS split, subscribed-vs-not, demographics, geography, cards/end-screen, vsChannelAverage). **This is the deepest data the agent has — and it's the gold here.**
- `get_channel_audience` — channel-wide demographics, top 25 countries, device split, traffic sources.
- `get_channel_revenue` — revenue / CPM / RPM / monetized playbacks / top 10 earning videos. Owner-tier OAuth required.

**Group `strategy`** (lines 408-487, dispatcher 852-1032)
- `list_competitors` — competitors tracked for the active channel + tier + last sync.
- `list_competitor_alerts` — viral competitor videos (≥2× their own median).
- `competitor_gap_analysis` — title keywords frequent in competitor top videos that the user never used.
- `get_comment_analysis` — cached AI sentiment/themes/objections/future-ideas/hook candidates for one of the user's videos.
- `list_outliers` — competitor outliers in active channel scope (3× their channel median, sorted by multiplier).
- `explain_outlier` — cached "what made it work" §9 levers + 2-3 sentence explanation per outlier.
- `generate_ideas` — 5-10 ideas grounded in channel context + outliers, rate-limited 1/5 min per channel.
- `list_format_patterns` — §4 structural title templates extracted from outliers + weekly trend points.

### D.2 What the agent CAN'T access (HAmo has mentioned wanting these)

Listing in priority order based on what showed up in the brief and the
methodology:

1. **Competitor transcripts** — there's a `transcripts` table and the
   agent can search the *user's own* transcripts (`search_my_transcripts`),
   but there's no tool to fetch a competitor's video transcript
   directly. Workaround: agent can call `fetch_transcript` (the
   public/free one) one video at a time — but it can't bulk-search
   across all competitor transcripts ("which competitors talked about
   topic X?"). This is the biggest gap for ideation: structural title
   patterns are easy, but knowing *what the competitor actually said
   about a topic* is invisible.

2. **Viral keywords / trending phrases from broader YouTube** —
   `youtube_trending` returns trending video titles by region, but
   there's no aggregation across them ("which phrases / N-grams are
   spiking right now"). `youtube_suggest` gives autocomplete on a
   *specified* seed query, not "what's hot". Topic Validator (from the
   brief, intentionally not yet built) would close this. Without it,
   the agent can only do reactive trend analysis when the user
   already names the topic.

3. **Thumbnail vision / image analysis** — Chat accepts image
   attachments (lines 580-633 in `src/app/chat/page.tsx`) and they
   ship as base64 to the provider, but there's NO dedicated tool that
   downloads a competitor's thumbnail and lets the agent describe its
   visual elements ("title overlay style, face crop, color palette").
   For a methodology that says *90% of your thumbnail should be the
   proven outlier format* (MENTOR_METHOD §7), this is a meaningful
   gap. The user has to manually screenshot and attach.

4. **Reddit signal / off-YouTube social signal** — the "External
   sources" field on /channel-info lets the user list subreddits etc.,
   but no tool reads from them. Exa `web_search` could in theory be
   instructed to focus on `site:reddit.com/r/...` but the agent has to
   know to do that — there's no dedicated `reddit_top_threads` tool.

5. **Suggested Videos from YT Studio** — `get_video_analytics`
   exposes "traffic sources" but not the *list of suggested videos
   that drove views*. From the methodology, knowing "my video got
   served next to competitor X's video" is high-signal — it's not in
   the bundle as currently built.

6. **TikTok / Instagram / shorts cross-platform signal** — none of
   this is wired, even though MCP-side tools for IG / TT exist in the
   broader Cowork environment (vidiq_ig_*, tiktok_*). The agent can't
   currently reach those from inside chat-tools.ts.

7. **The user's saved styles / formats library** — there's a
   `list_format_patterns` (§4 extracted templates), but there's no
   *user-curated* styles library that the agent reads from. From the
   brief: "Styles Library" was tagged coming-soon. Once shipped, the
   agent should be able to call `list_my_saved_styles` and use those
   as building blocks alongside the auto-extracted §4 templates.

8. **CTR / impressions split by thumbnail variant** —
   `get_video_analytics` is rich but doesn't include the
   impressions+CTR comparison across thumbnail variants tested. YT
   Studio surfaces this; the agent doesn't see it.

9. **Per-channel monetization status / RPM history** — `get_channel_revenue`
   returns aggregate revenue but doesn't return per-channel RPM trend.
   Less critical for ideation, but the user is multi-channel and may
   want "which channel's RPM is climbing".

### D.3 Single biggest gap (one-line answer to the brief)

**Competitor transcripts — bulk fetch + bulk search.** Title patterns
and thumbnail formats are surface-level; the *content* of the videos
the competitors are winning with is the real fuel for ideation, and
the agent currently can't read it without manual one-at-a-time
transcript fetches per video.

## Section E — Top 5 prioritized fixes

1. **A4 — Channel Info cross-channel data leak**
   Rationale: when the multi-channel summary table is the user's
   entry point, clicking through to a row that silently shows a
   *different* channel's audience+revenue numbers is a credibility
   bug. People will stop trusting any number on the page.

2. **A1+C2 — Settings discoverability + headline duplication**
   Rationale: the user can't easily reach Logs / Alerts / Import
   today. These are core admin tools. Surface them in the sidebar
   directly (or expand the sidebar Settings item into a nested set).
   At the same time, drop the duplicate h1 in SettingsLayout.

3. **D.3 — Competitor transcript tool**
   Rationale: this is the biggest *capability* gap, not a bug. The
   agent's reasoning quality jumps once it can read what competitors
   actually say about a topic — without it, every "topic gap" answer
   is title-keyword-based, which is the shallowest possible signal.

4. **B1+B2 — Dashboard de-duplication (cut ViewsOverTime + lower KPI strip)**
   Rationale: less surface area to maintain, less ambiguous numbers
   ("which subs count is right?"). The Dashboard becomes the
   high-signal page the user actually returns to.

5. **A2+A3+B3+B4 — Tutorial rewrite (consistency + cut filler)**
   Rationale: the tutorial is the first thing a new user sees, and
   right now it tells them three contradictory stories about what's
   shipped. The faster fix is also the smallest fix here — drop the
   "Coming soon" pills on shipped features, drop the page list (each
   page self-labels), keep methodology + quick-start, route quick-start
   step 4 to Chat (the actual ideation entrypoint).
